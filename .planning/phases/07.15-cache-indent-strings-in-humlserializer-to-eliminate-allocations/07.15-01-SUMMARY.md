# Phase 07.15 Plan 01: Cache Indent Strings in HumlSerializer Summary

**IndentCache static string[65] pre-computed at class load eliminates per-call new string(' ', depth * 2) allocations on the serializer hot path for depths 0-64**

## Accomplishments

- Replaced per-call `new string(' ', depth * 2)` in `Indent()` with O(1) static array lookup for depths 0-64
- Added `BuildIndentCache(64)` factory initialising 65 string entries at class load time
- Used unsigned-cast bounds check `(uint)depth < (uint)IndentCache.Length` to handle negative depth safely without extra branch
- Added `HumlSerializerAllocationTests.cs` with 3 tests: allocation reduction, deep-nesting correctness, and per-depth space count regression guard
- 721 tests pass across net8.0/net9.0/net10.0 (up from 718 - the 3 new allocation tests)

- `tests/Huml.Net.Tests/Serialization/HumlSerializerAllocationTests.cs` - 3 allocation/correctness tests using `GC.GetAllocatedBytesForCurrentThread`

## Decisions Made

- IndentCache covers indices 0-64 (65 entries), matching the `MaxRecursionDepth` default of 64 established in Phase 07.9. Typical documents never exceed this depth.
- Unsigned cast pattern `(uint)depth < (uint)IndentCache.Length` handles negative depth safely (wraps to large positive uint, falls to dynamic path).
- `BeLessOrEqualTo` is not defined in AwesomeAssertions 9.4.0. Allocation comparison rewritten as `BeLessThan(alloc1 + 512)` with a 512-byte noise margin to tolerate JIT/GC metadata jitter between measurements.
